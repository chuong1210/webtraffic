## Architecture decisions

- **AI in `ml/` only:** YOLO, ROI filter, tracker, counter live under `backend/app/ml/`; routes and services orchestrate only.
- **One active model:** Only one YOLO model loaded at a time; upload replaces file in `models_storage` and loads it.
- **WebSocket for real-time:** Single endpoint `/api/v1/stream/ws` streams base64 frame + detections + stats; frontend draws on canvas.
  (Note: later removed; system now uses REST + /stream/status only, no WebSocket.)
- **ROI:** Polygon as list of `[x, y]`; backend filters detections by centroid-in-polygon when ROI is active.

---

## Performance / baselines

- Backend: FPS capped by `MAX_FPS` (default 30); frame interval applied in stream worker.
- Frontend: No baseline captured; consider measuring WebSocket frame rate and canvas draw time if needed.

## Pain points (known)

- ROI coordinates from frontend are canvas-relative; scaling to video coordinates may be needed for different resolutions (currently backend accepts as-is; ROI filter uses integer points).
- No auth on API or WebSocket; suitable for demo/lab only.

### 2026-03-14 – Backend crash on import after removing CentroidTracker

**Symptom:** Backend không khởi động được, crash ngay khi import module với `ModuleNotFoundError: No module named 'sort_tracker'`.

**Expected:** Backend start bình thường; nếu thiếu tracker dependency thì báo lỗi rõ ràng lúc startup.

**Root cause:** `stream_service` singleton tạo `StreamService()` tại import-time, và trong `__init__` khởi tạo tracker ngay (`get_tracker('sort')`) → import fail trước khi lifespan startup check chạy.

**Fix:** Khởi tạo tracker **lazy** trong `StreamService` (để import không crash). Việc kiểm tra dependency được thực hiện trong `app.main` lifespan (fail-fast với thông báo cài đặt). Đồng thời `_get_sort_class()` raise ImportError với message rõ ràng khi thiếu `sort-tracker-py`.

**Affected files:** `backend/app/services/stream_service.py`, `backend/app/ml/tracker.py`, `backend/app/main.py`.

### 2025-03-14 – Duplicate ROI UI at bottom of screen (layout bug)

**Symptom:** ROI panel content (instructions, Draw/Undo/Clear/Apply buttons, FPS/Frame/ROI badges, 0 điểm, Inactive) appeared twice: once in the left sidebar (correct) and again at the bottom of the main area, causing overflow and a broken layout.

- Replaced `RoiOverlay` (full RoiDrawer) in the main area with `RoiCanvasOverlay` using the same state. Removed the unused `RoiOverlay` wrapper.

**Affected files:** `frontend/src/pages/Dashboard/Dashboard.tsx`, `frontend/src/components/RoiDrawer/RoiDrawer.tsx`, `frontend/src/components/RoiDrawer/index.ts`.

**Lessons:** When the same component is used in two places (sidebar + overlay), avoid rendering the full UI in both; split into “controls” and “canvas” and share state from the parent to prevent duplication and wrong layout. Responsive/layout issue was not due to viewport CSS but to duplicate DOM and flow.

### 2025-03-14 – Connect bị đứng (request không trả về)

**Symptom:** Bấm Connect trên dashboard thì giao diện “đứng” (loading mãi), không báo lỗi, không vào stream.

**Expected:** POST /stream/start trả về trong vài giây; nếu URL sai thì báo lỗi hoặc stream không có frame.

**Evidence:** Trong `stream_service.start()` có gọi `_probe_line_y(effective_url)`. Hàm này mở `cv2.VideoCapture(url)` và `cap.read()` ngay trên luồng xử lý HTTP. Với RTSP/URL chậm hoặc không tồn tại, OpenCV có thể block rất lâu (hoặc không timeout), nên request POST không bao giờ trả về, frontend kẹt ở `await startStream(...)` (setConnecting(true) mãi).

**Root cause:** Probe stream (mở capture + đọc 1 frame) chạy đồng bộ trong request handler, khiến HTTP response bị block.

**Fix:** Bỏ gọi `_probe_line_y(effective_url)` trong `start()`. Worker thread đã tự set `_counting_line_y` ở frame đầu (khi `_counting_line_y is None`). API trả về ngay; worker mở stream ở background — nếu mở thất bại thì không có frame, `stream_active` vẫn false.

**Affected files:** `backend/app/services/stream_service.py`.

**Risk:** Nếu URL sai, user vẫn thấy “Stream đã kết nối” nhưng không có hình (worker mở thất bại). Có thể cải thiện sau bằng cách báo lỗi khi worker không mở được (ví dụ qua /stream/status hoặc WebSocket).

### 2025-03-14 – POST /stream/start với YouTube treo (HTTP không trả về)

**Symptom:** Gọi POST /stream/start với URL YouTube thì request không trả về, frontend treo "Connecting...", client timeout.

**Cause:** resolve_youtube_url() chạy đồng bộ trong request handler; yt-dlp có thể chạy lâu (Windows/network) → HTTP không trả lời. Ngoài ra start() gọi stop() trước khi spawn thread; stop() gọi _thread.join(timeout=4) → nếu có stream cũ, request bị block tới 4s.

**Fix (non-blocking end-to-end):**
- **Backend:** start(url) không làm gì nặng trên request thread: chỉ clear error, spawn **một** daemon thread chạy _do_start_with_stop(url). Trong thread: gọi stop() (join worker cũ), rồi _do_start(url) (validate YouTube, resolve_youtube_url(), start worker). POST /stream/start trả **202 Accepted** ngay với message "Stream đang kết nối (YouTube có thể mất vài giây)...". Khi resolve/connect lỗi, set _stats.stream_error và _last_error; GET /stream/status trả về { active, fps, frame_count, error }.
- **Frontend:** Sau khi startStream() thành công (202), hiện toast "Stream đang kết nối...", bật streamOn. Poll GET /stream/status mỗi 1.5s tối đa 25s: nếu status.error thì toast lỗi và set streamOn = false; nếu status.active thì dừng poll.
- **Models:** VehicleStats có stream_error: str = ""; stream_routes stream_status trả error từ stream_error.

**Affected files:** `backend/app/services/stream_service.py`, `backend/app/api/stream_routes.py`, `backend/app/models/detection_model.py`, `frontend/src/services/api.ts` (StreamStatus, getStatus), `frontend/src/pages/Dashboard/Dashboard.tsx` (poll sau Connect), `frontend/src/types/detection.ts` (stream_error optional).

**Lessons:** Để HTTP không bao giờ treo: mọi việc tốn thời gian (resolve, open stream, stop + join) phải chạy trong background thread; request handler chỉ spawn thread và return 202. Frontend cần poll status (hoặc WebSocket có error) để hiển thị lỗi khi resolve/connect thất bại.

**Risk:** User phải **restart backend** sau khi pull code mới thì 202 mới có hiệu lực; nếu vẫn chạy process cũ thì request vẫn có thể block.

**Lessons:** Khi bỏ WebSocket hoặc đổi giao thức realtime, phải đảm bảo có đường thay thế truyền payload đầy đủ (frame + detections + stats), và cập nhật hook/state ở frontend tương ứng. ### 2026-03-14 – Nhãn YOLO không khớp với dataset custom (toàn báo car)

**Symptom:** Mô hình YOLO custom được train với YAML:
`names: ['pedestrian', 'bicycle', 'motorbike', 'bus', 'truck', 'container truck', 'car']`, nhưng khi chạy demo web thì bounding box trên UI gần như chỉ hiển thị nhãn `car` (và một số nhãn không đúng với loại phương tiện).

**Evidence:**
- Trong `backend/app/ml/yolo_model.py` có constant `VEHICLE_CLASSES` map theo COCO: `{0:'person',1:'bicycle',2:'car',3:'motorcycle',5:'bus',7:'truck'}`.
- Hàm `predict()` lọc `cls_id` theo `VEHICLE_CLASSES` và gán `class_name = VEHICLE_CLASSES[cls_id]`.
- Với dataset custom:
  - class 0 = pedestrian → map thành `"person"` (tạm ổn),
  - class 1 = bicycle → `"bicycle"` (ổn),
  - class 2 = motorbike nhưng bị map thành `"car"`,
  - class 3 = bus nhưng bị map thành `"motorcycle"`,
  - class 4,5,6 (truck, container truck, car) **bị loại bỏ hẳn** vì không có trong `VEHICLE_CLASSES`.
- Kết quả: nhiều loại phương tiện bị dán nhãn sai hoặc không được vẽ, khiến người dùng thấy gần như toàn `car`.

**Root cause:** `VEHICLE_CLASSES` giả định mô hình là COCO mặc định; với mô hình custom, `cls_id` theo YAML riêng nên mapping cố định này sai hoàn toàn. Code không lấy `model.names` từ ultralytics, cũng không hỗ trợ đủ số lớp trong dataset.

**Fix:**
- Bỏ constant `VEHICLE_CLASSES`.
- Trong `YOLOModel.load()`, sau khi `YOLO(path)`:
  - Đọc `self._model.names` (dict hoặc list) và build `_class_names: dict[int,str]` từ đó để phản ánh đúng YAML train.
- Trong `predict()`:
  - Không filter theo `VEHICLE_CLASSES` nữa; tạo `RawDetection.class_name` bằng `self._class_names.get(cls_id, str(cls_id))`.
- Reset `_class_names` về `{}` khi `unload()`.

**Affected files:** `backend/app/ml/yolo_model.py`.

**Lessons:** Khi dùng YOLO custom với YAML riêng, luôn lấy mapping lớp từ chính mô hình (`model.names`) thay vì hard-code COCO IDs. Nếu muốn đếm/hiển thị một subset (chỉ vehicle chẳng hạn), thì cũng nên filter bằng tên lớp từ YAML (ví dụ chứa 'car', 'truck', 'bus', 'motorbike') chứ không dựa vào chỉ số COCO mặc định.

### 2026-03-14 – Sidebar không scroll, card cảm giác bị đè (zoom 100%)

**Symptom:** Ở zoom 100% người dùng thấy các card trong sidebar (Model, RTSP, ROI, Cài Đặt) dồn sát nhau, cảm giác bị \"đè\" và không thể scroll sidebar; khi zoom 90% thì trông bình thường.

**Evidence:** Root layout dùng `<div className=\"flex flex-col h-screen overflow-hidden\">`; body row là `<div className=\"flex flex-1\">` (lúc đầu thiếu `min-h-0`). Sidebar là flex child rộng cố định, khi nội dung cao hơn viewport thì flex row đẩy chiều cao cha lên nhưng do root overflow hidden nên không có scrollbar, dẫn đến cảm giác chồng lấn. Sau fix: body row dùng `min-h-0`, sidebar có `overflow-y-auto`, main cũng có `min-h-0`.

**Root cause:** Flex container chính không cho phép con co lại theo chiều dọc (`min-height` mặc định), và sidebar thiếu `overflow-y-auto`. Khi nội dung cao hơn viewport (nhất là 100–125% zoom), layout không có vùng scroll nội bộ, khiến card bị dồn và trông như chồng lên nhau.

**Fix:** 
- Thêm `min-h-0` cho body row và main: `<div className=\"flex flex-1 min-h-0\">`, `<main className=\"flex flex-1 min-w-0 min-h-0 overflow-hidden\">`.
- Đặt `overflow-y-auto` cho sidebar: `<aside className=\"w-80 ... overflow-y-auto bg-white flex flex-col gap-2 p-3\">`.
- Đảm bảo `SideCard` và nội dung bên trong không dùng height cố định, chỉ dùng padding/gap.

**Affected files:** `frontend/src/pages/Dashboard/Dashboard.tsx`.

**Lessons:** Với bố cục full-screen dùng flex, luôn dùng `min-h-0` cho các flex child dọc cần chứa vùng scroll, và gán `overflow-y-auto` trực tiếp cho vùng đó (sidebar, stats). Tránh rely vào page scroll khi root đang `overflow-hidden`, đặc biệt vì zoom khác nhau sẽ làm nội dung vượt viewport.

### 2026-03-14 – Tất cả bounding box đều hiển thị ID0 (SORT tracker)

**Symptom:** Trong giao diện web mới (FastAPI + React), mọi bounding box đều hiển thị `ID0` (ví dụ `car ID0 58%`, `motorbike ID0 88%`) dù đang bật tracker SORT/DeepSORT.

**Evidence:**
- Backend dùng lớp `SortTrackerAdapter` trong `backend/app/ml/tracker.py`, map output của thư viện SORT thành `Track.track_id`.
- Adapter giả định output có dạng `[x1, y1, x2, y2, id]`. Tuy nhiên một số implementation SORT (6 cột) trả về `[x1, y1, x2, y2, score, id]`.
- Trong code cũ, `track_id` luôn lấy từ `row[4]`. Nếu cột 4 thực ra là confidence (0–1), việc `int(row[4])` sẽ luôn cho `0`, dẫn tới toàn bộ track có `track_id = 0` và UI vẽ `ID0` cho mọi object.

**Root cause:** Giả định sai về sơ đồ cột trong output của SORT – luôn dùng cột 4 làm `track_id` kể cả với những thư viện trả về 6 cột `[x1,y1,x2,y2,score,id]`, khiến score bị cast thành ID=0.

**Fix applied:**
- Trong `SortTrackerAdapter.update()`:
  - Sau khi `out = np.atleast_2d(out)`, đọc `n_cols = out.shape[1]`.
  - Nếu `n_cols >= 6` → lấy `track_id` từ `row[-1]` (cột cuối, tương thích với format `[x1,y1,x2,y2,score,id]` và các format mở rộng).
  - Nếu không (5 cột) → giữ nguyên hành vi cũ: `track_id` từ `row[4]` (đúng với bản SORT gốc của abewley).
  - Hàm `_prune_ids` cũng được cập nhật để lấy ID từ cột cuối khi `n_cols >= 6`.

**Affected files:** `backend/app/ml/tracker.py`.

**Lessons / remaining risks:**
- Khi wrap thư viện bên ngoài (SORT/DeepSORT), luôn kiểm tra kỹ format output (số cột, vị trí ID) và viết adapter để xử lý nhiều biến thể thông dụng (5 hoặc 6 cột) thay vì hard-code một layout duy nhất.
- Nếu gặp implementation SORT khác (ví dụ output MOT với 8+ cột), adapter hiện tại có thể vẫn không khớp – khi đó nên bổ sung nhánh parse riêng hoặc ép dùng đúng package đã kiểm thử (ví dụ sort của abewley hoặc sort-tracker-py với format rõ ràng).
